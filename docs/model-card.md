# IDBI MSME HealthLens — ML Model Card

> Generated by `npm run train`. Synthetic data only — NOT production-calibrated.

## Purpose

A transparent logistic scorecard that estimates an MSME's **viability** (1 − probability of default) from consented alternate data. It is a secondary, explainable AI signal on the Health Card; it never overrides the rule-based HealthScore or the BRE decision.

## Label

`synthetic_default` — a leakage-free, stochastic Bernoulli draw from each MSME's hidden `latent_default_propensity`. No model feature is a deterministic function of the label.

## Data

- Synthetic population: 1000 MSMEs (deterministic seed).
- Train: 700 · Held-out test: 300.
- Train default rate: 23.9%.

| Metric | Logistic (champion) | GBM (challenger) |
| ------ | ------------------- | ---------------- |
| AUC    | 0.7449              | 0.7498           |
| KS     | 0.3609              | 0.4033           |
| Gini   | 0.4898              | 0.4997           |
| Trees  | —                   | 200              |

The **champion** stays the monotonic logistic scorecard (exact additive
explanations, sign-constrained, final control). The **GBM challenger** is a
non-monotonic benchmark run alongside it to confirm the scorecard is not leaving
material signal on the table. A **calibration curve** and a **band → expected-PD**
table are surfaced on the Governance screen; in a pilot the challenger only gets
promoted after backtesting and calibration on IDBI's anonymised portfolio.

## Features and fitted direction

| Feature                                          | Weight (standardised) | Effect                |
| ------------------------------------------------ | --------------------- | --------------------- |
| GST filing punctuality (`gstOnTimeRatio`)        | 0.000                 | ≈ neutral (redundant) |
| Cash-flow volatility (`bankCreditVolatility`)    | 0.070                 | ↑ default             |
| EMI/cheque bounces (`emiBounceCount`)            | 0.256                 | ↑ default             |
| GST-vs-bank mismatch (`gstBankTurnoverGap`)      | 0.068                 | ↑ default             |
| UPI refund ratio (`upiRefundRatio`)              | 0.103                 | ↑ default             |
| Debt-service ratio (`debtServiceRatio`)          | 0.000                 | ≈ neutral (redundant) |
| Worst DPD (`dpdDays`)                            | 0.081                 | ↑ default             |
| EPFO contribution gaps (`epfoMissingMonthRatio`) | 0.000                 | ≈ neutral (redundant) |
| Buyer concentration (`gstConcentration`)         | 0.456                 | ↑ default             |
| Digital-payments momentum (`upiTrend`)           | -0.099                | ↓ default             |
| Power-consumption momentum (`powerTrend`)        | 0.000                 | ≈ neutral (redundant) |
| Power-vs-turnover mismatch (`turnoverPowerGap`)  | 0.163                 | ↑ default             |

> **Electricity / power-consumption signals** (`powerTrend`, `turnoverPowerGap`) are
> retained as features but show ~0 weight on the full population because they are
> statistically redundant with the cash-flow and GST-vs-bank signals the model has
> already learned. They remain decision-relevant through (a) the rule-based
> **Power & operations** sub-score (10% of the primary HealthScore), (b) the
> power-vs-turnover fraud flag, and (c) thin-file applicants with no GST/bureau,
> where power is the only confirmation of real operating activity. A sign-constrained
> scorecard should not be forced to load a redundant feature for appearances.

## Explainability

For this linear model the per-feature contribution is **exact** (weight × standardised value) — an honest additive explanation. The Explainability screen's contribution bars explain `mlProbabilityProxy` only, never the HealthScore.

## Limitations

- Trained on synthetic, schema-faithful data; metrics are illustrative, not production performance.
- In a pilot, refit and calibrate against IDBI's anonymised MSME portfolio with WoE/optimal binning and stronger models (LightGBM), keeping BRE rules as the final control.

## Governance

- Owner: HealthLens credit-decisioning team.
- Final decision authority: IDBI (human-in-the-loop). HealthLens is decision-support only.
- Refit cadence: each dataset/seed change re-runs `npm run train` deterministically.
